# 백엔드 ↔ AI 에이전트 연동 협의 명세서 (통합본)

> Spring Boot ↔ FastAPI 통신에서 **두 팀이 협의·합의해야 하는 모든 것**을 한 문서에 담음.
> API 계약 · DB 연동 · 인증/네트워크 · 타임아웃 · 회복 · 관측성 · 합의 체크리스트.

| 항목 | 값 |
|---|---|
| 통신 | Spring Boot → FastAPI, **동기 JSON** (SSE 미사용) |
| 엔드포인트 | `POST /api/v1/chat` |
| 포트 | 8001 (사설망 내부, 공용 노출 금지) |
| 버전 | v2.1 (백엔드 v2.0 기반 + DB/합의 통합) |
| 기계판독 스키마 | `chat_v2_schema.json` (요청·응답 JSON Schema) |

---

## 0. 책임 분리

| 시스템 | 책임 |
|---|---|
| **Spring Boot** | 사용자 인증, 대화방·이력 DB, 컨텍스트 조립(messages), FastAPI 동기 호출, 결과 영속화, 클라이언트 응답 |
| **FastAPI (AI)** | Stateless RAG + 검증 추론. (질문+이력+기업) → **답변·출처·거시·검증·분기신호를 단일 JSON**으로 반환. 대화 상태 미보관 |

## 2. 요청 (Spring → FastAPI)

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `roomId` | integer | ✅ | 대화방 ID (응답 echo) |
| `userSeq` | integer | ✅ | 사용자 시퀀스 (로그) |
| `companyContext` | object | ✅ | 대화 대상 기업 (한 방 = 한 기업, 변경 불가) |
| `companyContext.corpCode` | string | ✅ | DART 고유번호 8자리 — RAG 검색 범위 키 |
| `companyContext.corpName` | string | ✅ | 기업명 |
| `messages` | array | ✅ | 멀티턴. 시간순, **마지막=`role:user`(현재 질문)** |
| `messages[].role` | enum | ✅ | `user` \| `assistant` |
| `messages[].content` | string | ✅ | 본문 |

> AI는 `messages` 중 **최근 10개만** 단기 메모리로 사용(토큰 안전장치). 마지막 user=질문, 나머지=이력.

```json
{
  "roomId": 142,
  "userSeq": 12345,
  "companyContext": {"corpCode": "00126380", "corpName": "삼성전자"},
  "messages": [
    {"role": "user", "content": "영업이익 알려줘"},
    {"role": "assistant", "content": "..."},
    {"role": "user", "content": "작년보다 얼마나 늘었어?"}
  ]
}
```

## 3. 응답 (FastAPI → Spring)

모든 키 항상 존재(값 채움/`null`). **추론 실패도 HTTP 200 + `error`**, 시스템 실패만 4xx/5xx.

| 필드 | 타입 | 설명 |
|---|---|---|
| `roomId` | integer | 요청 echo |
| `intent` | string\|null | `qa`/`summary`/`macro`/`smalltalk`/`out_of_scope` |
| `answerText` | string\|null | 답변 본문 (완전 실패 시 `null`) |
| `sourceContent` | string\|null | 인용 근거 텍스트(§4 포맷). 근거 없으면 `null` |
| `macroSnapshot` | string\|null | 거시지표 텍스트(환율·기준금리·국고채3년·KOSPI). 미사용 `null` |
| `sources` | array\|null | 구조화 출처(프론트 카드용) |
| `sources[].rceptNo` | string | 공시 접수번호 |
| `sources[].reportNm` | string | 공시명 |
| `sources[].rceptDt` | string | 접수일 YYYYMMDD |
| `sources[].sectionTitle` | string | 인용 섹션 |
| `sources[].quote` | string | 인용 본문 |
| `sources[].score` | number | 유사도 0~1 |
| `outOfScope` | boolean | true → 다른 회사 질문 → 프론트 팝업 |
| `detectedCompany` | string\|null | outOfScope 시 감지 회사 |
| `needsClarification` | boolean | true → `answerText`가 되묻는 질문 |
| `verification` | object\|null | `{verdict, groundedScore}` (smalltalk/out_of_scope/실패 시 null) |
| `verification.verdict` | string | `pass`/`partial`/`fail` |
| `verification.groundedScore` | number | 근거 충실도 0~1 |
| `error` | object\|null | `{code, message, retriable}` (완전 실패 시) |

### 케이스별 예시
**정상(qa)**
```json
{
  "roomId":142,"intent":"qa",
  "answerText":"연결 영업이익은 전기 32.7조 → 당기 43.6조, 10.875조 증가...",
  "sourceContent":"[20260310002820 / 삼성전자 2025 사업보고서 재무제표]\n영업이익 — 당기 43,601,051,000,000원...",
  "macroSnapshot":null,
  "sources":[{"rceptNo":"20260310002820","reportNm":"2025 사업보고서 재무제표","rceptDt":"20260310","sectionTitle":"재무제표 주요계정","quote":"영업이익 — 당기 43,601,051,000,000원...","score":1.0}],
  "outOfScope":false,"detectedCompany":null,"needsClarification":false,
  "verification":{"verdict":"pass","groundedScore":1.0},"error":null
}
```
**다른 회사(out_of_scope)** — `answerText`=팝업문구, `outOfScope`:true, 나머지 근거 null
**되묻기(needsClarification)** — `answerText`=되묻는 질문, `needsClarification`:true, 근거 null
**실패** — `answerText`:null, `error`:{"code":"LLM_TIMEOUT","message":"...","retriable":true}

## 4. `sourceContent` 포맷
```
[<rceptNo> / <corpName> <reportNm>]
<인용 본문>

[<rceptNo> / <corpName> <reportNm>]
<인용 본문>
```
청크 간 `\n\n`. DB 저장은 이 문자열, 프론트 카드는 `sources[]`.

## 5. 값 정의

### 5.1 intent → 출처/검증 유무
| intent | 출처 | 검증 |
|---|---|---|
| `qa`/`summary`/`macro` | ✅ | ✅ |
| `smalltalk`/`out_of_scope` | ❌ | ❌ |

### 5.2 verification.verdict (groundedScore로 시스템이 확정)
| groundedScore | verdict |
|---|---|
| ≥ 0.7 | pass |
| 0.4 ~ 0.7 | partial |
| < 0.4 | fail (CRAG 재검색 1회 후에도 실패. 답은 반환, 신뢰도 낮음 표기) |

### 5.3 error.code / HTTP status
| error.code | 상황 | retriable |
|---|---|---|
| `EMBEDDING_FAILED`·`VECTOR_SEARCH_FAILED` | 임베딩·검색 실패 | true |
| `LLM_TIMEOUT`·`LLM_API_ERROR`·`RATE_LIMITED` | LLM 호출 문제 | true |
| `CONTEXT_OVERFLOW`·`CONTENT_FILTERED` | 컨텍스트 초과·필터 거부 | false |
| `INVALID_REQUEST` | 요청 스키마 오류(HTTP 400) | false |
| `INTERNAL_ERROR` | 미분류 | true |

| HTTP | 상황 |
|---|---|
| 200 | 정상 + 추론실패(body.error) |
| 400 | 요청 스키마 오류 (messages 마지막이 user 아님 등) |
| 503 / 504 | 서비스 다운 / 게이트웨이 타임아웃 |

## 6. DB 연동 (협의 필요)

### 6.1 소유권
| 저장소 | 소유 | 백엔드 접근 |
|---|---|---|
| 대화·세션·이력 DB | **백엔드** | 직접 소유 |
| 공시 코퍼스(Chroma 벡터DB) | **AI** | ❌ (AI가 검색해 결과만 응답) |
| 거시 캐시(SQLite) | **AI** | ❌ |

→ **AI는 대화 내용을 저장하지 않음.** 이력 영속화는 백엔드 책임.

### 6.2 대화 이력 저장 — AI 응답 → 백엔드 DB 컬럼 (제안)
백엔드 테이블(예 `tb_qa_history`)에 저장 권장:

| 저장 컬럼(예) | ← AI 응답 필드 | 비고 |
|---|---|---|
| `room_id` | `roomId` | |
| `role` / `content` | 사용자질문 / `answerText` | 턴 본문 |
| `source_content` | `sourceContent` | 인용 근거 텍스트 그대로 |
| `macro_snapshot` | `macroSnapshot` | 거시 텍스트 (있을 때) |
| `intent` | `intent` | 분석용(선택) |
| `verdict` / `grounded_score` | `verification.verdict` / `.groundedScore` | 신뢰도(선택) |
| `created_at` | (백엔드 시각) | |

- 구조화 `sources[]`는 보통 저장 불필요(표시용). 필요 시 JSON 컬럼.
- **`error != null`(추론 실패) 응답은 이력에 저장하지 않음.**
- 테이블 DDL은 백엔드 자유 설계 (위는 "담아두면 좋은 데이터" 가이드).

### 6.3 검색 가능 기업(corpCode) 동기화
- **Spring 선택 가능 회사 목록 = AI 코퍼스 적재 기업**과 일치해야 함.
- 현재 적재: **삼성전자 `00126380`, 현대자동차 `00164742`** (3개년 공시).
- 새 기업 추가: **AI가 먼저 코퍼스 적재 → 백엔드 목록 추가** (미적재 기업은 검색 비어 "확인 불가"만 나옴).
- 식별자 = **DART 8자리 corpCode 통일** (백엔드가 회사명↔corpCode 매핑 보유).

### 6.4 AI 가동 데이터 상세 (ChromaDB + 에이전트 저장)

ChromaDB는 Vector DB라 RDB식 테이블 정의가 아니라 **벡터 데이터 구조 + 컨텍스트 관리 방식**으로 기술.

#### (1) Collection 명세
- 명명규칙 **`corpus_<corpCode>`** = 기업 1개당 1개 컬렉션 (목적: 그 기업의 공시 코퍼스 = RAG 검색 대상)
- 현재: `corpus_00126380`(삼성전자, 60,119청크), `corpus_00164742`(현대자동차, 77,505청크)
- **회사 범위 제한 = 컬렉션 분리**로 구현 (질문은 해당 기업 컬렉션만 검색)

#### (2) Embedding 모델 / 차원
- 모델 **`text-embedding-3-small`**, **1536 dim** / 검색 점수 `score = 1 − distance`
- 적재·검색 **동일 모델 필수**. 차원(1536)이 벡터 스키마 크기 결정.

#### (3) Metadata 스키마 (★ 필터·출처 핵심)
청크(레코드) = `id` + `document`(본문) + `embedding`(1536d) + `metadata`.
id 규칙 `<rceptNo>-c-<순번>` (예 `20260608800918-c-0000`).

| metadata key | type | 용도 |
|---|---|---|
| `corp_name` | string | 기업명 |
| `rcept_no` | string | 공시 접수번호 (출처 추적) |
| `rcept_dt` | string (YYYYMMDD) | 날짜 범위 필터 / 최신 정렬 |
| `report_nm` | string | 공시명 |
| `pblntf_ty` | string | 공시유형(A정기/B주요사항/C발행/D지분/F외부감사/I거래소) |
| `section_title` | string | 인용 섹션 |
| `order` | integer | 문서 내 청크 순번 |

- **필터링 방식**: 회사=컬렉션 분리 / 기간("작년·2024년")=`rcept_dt` 범위 / "최근"=`rcept_dt` 최신 정렬.
- 응답 `sources[]`·`sourceContent`는 이 메타데이터에서 생성.

#### (4) 단기 메모리 (Short-term / 프롬프트 히스토리) — ★중요
- **AI는 대화 기록을 자체 저장하지 않음 (Stateless).** FastAPI 서버 메모리·Redis에 세션 보관 ❌.
- 단기 메모리 = **백엔드가 매 요청 `messages[]`로 전달** (원본 저장은 백엔드 대화 DB). AI는 그중 **최근 10개 메시지만** 프롬프트에 사용.
- 효과: 수평 확장 안전(어느 인스턴스가 받아도 동일 동작), 세션 동기화 이슈 없음.
- → 즉 **Chat History 저장소 = 백엔드(PostgreSQL 등)**. FastAPI/Redis 공유 없음.

#### (5) 에이전트 상태(State) 관리
- 프레임워크 = **CrewAI** (LangChain/LangGraph 아님).
- **에이전트 상태·스텝을 영속화하지 않음.** 한 요청 = 단발 파이프라인(Router→검색→QA→Verification, fail 시 CRAG 1회) 실행 후 종료.
- 되묻기·CRAG 재시도는 **한 요청 내부**(또는 다음 요청의 `messages`)로 처리 — 별도 state store(예: LangGraph checkpoint) 없음.

#### (6) SQLite — AI가 영속화하는 유일한 데이터 (대화와 무관)
| 테이블 | 컬럼 | 용도 |
|---|---|---|
| `macro_cache` | `as_of`(PK, YYYYMMDD), `payload`(JSON) | 거시 스냅샷 날짜별 캐시 (활성) |
| `analyses`·`chat_turns` | (레거시) | 단건분석 경로 — 챗 미사용 |

#### (7) 운영 메모
- 컬렉션 추가 = 새 기업 적재(`ingest_corpus.py`, corpCode별 `corpus_<corpCode>` 생성)
- 청킹 파라미터 변경 = 해당 코퍼스 **전체 재적재**(검색 일관성)
- 적재·검색 **동일 임베딩 모델** 고정

## 7. 인증·네트워크

- FastAPI는 **사내 사설망**(Docker/VPC/K8s ClusterIP) 내부에만 노출, 공용 인터넷 금지.
- **앱 레벨 인증 헤더 없음** (`X-Internal-Api-Key` 등 미사용). 네트워크 격리로 신뢰.
- AI는 수신한 `userSeq`·`roomId`를 신뢰(백엔드가 사전 인증 완료).

## 8. 타임아웃

| 항목 | 값 | 비고 |
|---|---|---|
| Connect Timeout | 10초 | 연결 실패 빠른 차단 |
| Write Timeout | 30초 | 대용량 messages 전송 마진 |
| **Response Timeout** | **90초** | 검색+추론+검증(+CRAG 재검색·거시결합) 전체 여유 |
| (AI 내부) LLM 호출 | 60초 | 초과 시 `LLM_TIMEOUT` |

## 9. 회복·재시도 정책

- **자동 재시도 없음** (LLM 비결정·토큰 비용). 
- 응답 `error != null` 시: 백엔드는 매핑 코드로 클라 중계 + **이력 미저장**.
- HTTP 4xx/5xx 시: 백엔드 일반 에러 처리 → 클라 `UPSTREAM_UNAVAILABLE` 등.
- `error.retriable=true` 면 클라 측 1회 재시도 권장(backoff 1→3초, 동일 messages).
- 검증 `fail` = 답 반환 + `verdict:fail`(거절 아님) → 프론트 "신뢰도 낮음" 표시.

---

## 10. 관측성

- 백엔드가 `X-Trace-Id`(UUID v4) 생성 → 요청 헤더 → AI가 동일 ID 로그 출력 + 응답 echo.
- 양 시스템 로그를 동일 traceId로 추적.

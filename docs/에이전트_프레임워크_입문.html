LLM 에이전트 프레임워크 입문

개발은 해봤지만 이 영역은 처음인 분을 위한 가벼운 지도 — LangChain · LangGraph · PydanticAI (+CrewAI 번외)

왜 프레임워크? LangChain LangGraph PydanticAI 번외: CrewAI 헷갈리는 것들 한눈 비교 언제 무엇을? 용어 사전

0. 왜 "프레임워크"가 필요할까?

LLM을 쓰는 건 사실 API 한 번 호출이 전부입니다. 그런데 실무에선 한 번으로 안 끝나요.

질문→ 관련 문서 검색(RAG)→ 도구 사용(계산·API)→ 여러 단계 추론→ 결과를 정해진 형식으로→ 검증

이런 여러 단계 · 도구 · 검색 · 구조화 출력 · 상태 관리 · 여러 에이전트 협업을 매번 직접 짜면 코드가 금방 엉킵니다. 프레임워크는 이 반복 작업을 정리해 주는 도구예요. 세 가지가 각각 다른 부분을 잘합니다.

🧭 한 줄 지도 — LangChain = "부품 상자(레고)", LangGraph = "순서도를 코드로", PydanticAI = "타입 안전한 에이전트", CrewAI = "역할극 팀".

1. LangChain 부품 상자 / 통합

🧱 비유: 레고 부품 상자. LLM·프롬프트·검색기·도구·메모리 같은 부품을 표준 인터페이스로 제공하고, 이를 체인(chain)으로 이어 붙인다.

핵심 개념

  • 컴포넌트: LLM 래퍼, 프롬프트 템플릿, 임베딩, 벡터스토어, 도구(tool), 메모리 등 — 공급자가 달라도 같은 방식으로 사용.
  • 체인(Chain): 컴포넌트를 순서대로 연결한 파이프라인. (예: 프롬프트 → LLM → 출력파서)
  • 엄청난 통합(integration) 생태계 — 수백 개 DB·모델·도구 커넥터.
장점 · 통합이 가장 많음 · 빠른 프로토타입 · RAG 부품이 풍부
단점 · 추상화가 많아 무겁고 "마법" 같음 · 버전 변화 잦음 · 복잡한 흐름엔 부족
🟡 언제: 여러 데이터·모델을 빠르게 붙여 RAG/프로토타입을 만들 때. "부품을 가져다 조립"이 필요할 때.

2. LangGraph 흐름 제어 / 오케스트레이션

🗺️ 비유: 순서도(flowchart)를 코드로. 작업을 노드로, 연결을 엣지로 그리고, 상태(state)를 들고 돌아다닌다. (LangChain 팀이 만듦)

핵심 개념

  • 노드(Node) = 한 작업(LLM 호출·검색·도구). 엣지(Edge) = 다음 노드 결정(조건부 분기 가능).
  • 상태(State) = 질문·검색결과·답·에러 등 누적 데이터. 노드가 읽고 갱신.
  • 사이클(반복)·병렬·체크포인트(중간 저장)·사람 개입(human-in-the-loop) 지원.
router→ (조건) 검색→ 작성→ 검증→ 실패면 ↺ 재시도
장점 · 복잡한 제어흐름(분기·반복·병렬)에 강함 · 부분실패 격리 · 관찰·영속성 · 생태계 인지도 최고
단점 · 단순 작업엔 과함 · 상태 타입 검증은 느슨(런타임 자동검증 약함)
🟦 언제: 여러 단계·조건분기·여러 에이전트가 협력하는 복잡한 워크플로우. "흐름 자체가 복잡할 때".

3. PydanticAI 타입 안전 / 구조화 출력

📐 비유: 규격이 보장되는 자판기. "이 형식(Pydantic 모델)으로 내놔"라고 하면 반드시 그 형식으로 검증해서 주고, 어긋나면 자동 재시도. (Pydantic 팀이 만듦, 2025 V1)

핵심 개념

  • 구조화 출력이 1급 시민: Agent(output_type=내Pydantic모델) → 결과를 Pydantic으로 검증 + 실패 시 자동 재시도.
  • 타입 안전: 파이썬 타입힌트 기반 → IDE 자동완성·정적 검사·런타임 검증.
  • 멀티에이전트 3패턴: ① 위임(에이전트를 도구처럼 호출) ② 핸드오프(파이썬 코드로 순차 위임) ③ pydantic-graph(LangGraph식 그래프).
  • Logfire 관찰성 내장.
# 가장 단순한 형태 — "구조화된 답"을 타입 안전하게
from pydantic import BaseModel
from pydantic_ai import Agent

class Answer(BaseModel):
    intent: str
    text: str

agent = Agent("openai:gpt-4o-mini", output_type=Answer)
res = agent.run_sync("이 공시 요약해줘")
print(res.output.intent, res.output.text)  # 검증된 Answer 객체
장점 · 코드량 최소 · 타입 안전+자동 검증/재시도 · Pydantic 쓰는 코드에 자연스러움 · 가벼움
단점 · 신생(인지도 LangGraph보다 낮음) · 초복잡 흐름은 pydantic-graph 필요
🟥 언제: 구조화된 결과·타입 안전이 중요하고 흐름은 그리 복잡하지 않을 때. 이미 Pydantic을 쓰는 코드베이스.

4. 번외 — CrewAI 역할극 팀

🎭 비유: 역할극 팀. 에이전트마다 역할(role)·목표(goal)를 주고 crew로 묶어 협업시킨다.

"리서처 → 작가 → 검수자" 처럼 역할을 나눠 팀으로 일을 시키는 직관적 추상화. 프로토타입을 가장 빨리 만들 수 있다는 게 매력. 다만 세밀한 흐름 제어·관찰·타입 안전은 약하고 의존성이 무겁다. (별도 오케스트레이션 없이 단일 에이전트 래퍼로만 쓰면 "그냥 LLM 호출"과 다를 바 없어져 가치가 떨어짐 — 자주 하는 실수.)

🟧 언제: 역할 기반 협업 데모를 빠르게 만들 때. 단, 제어가 중요하면 LangGraph/PydanticAI가 낫다.

5. 번외 — 자주 헷갈리는 것들

혼동 포인트정리
LangChain vs LangGraph같은 팀. LangChain = 부품·통합 라이브러리, LangGraph = 그 위의 흐름 오케스트레이션(상태그래프). 보통 함께 씀.
LlamaIndexRAG·데이터 인덱싱 특화 프레임워크(문서 요약 인덱스 등). "검색/인덱싱"에 강점 — LangChain과 자주 비교.
OpenAI Agents SDKOpenAI 공식 경량 에이전트 SDK. 단순·OpenAI 중심.
Microsoft AutoGen여러 에이전트가 대화하며 협업하는 멀티에이전트 프레임워크.
litellm프레임워크 아님. 여러 LLM 공급자(OpenAI·Anthropic 등)를 한 인터페이스로 호출하는 어댑터. (CrewAI 내부에서도 사용)
"에이전트 vs 체인 vs 워크플로우"체인=정해진 순서, 에이전트=LLM이 다음 행동을 스스로 결정(도구 선택 등), 워크플로우/그래프=분기·반복 있는 흐름. 경계는 흐릿하게 섞임.
RAG는 프레임워크인가?아니다. RAG = "검색해서 찾은 근거로 답하는 방식". 위 프레임워크들로 구현하는 패턴.

6. 한눈 비교

프레임워크한 줄 정체성강점약점언제
LangChain부품 상자(레고)통합 풍부·빠른 조립무겁고 추상화 많음RAG·프로토타입
LangGraph순서도를 코드로복잡 흐름·분기·병렬·인지도단순 작업엔 과함복잡 워크플로우
PydanticAI타입 안전 에이전트구조화 출력·검증·경량신생·초복잡 흐름은 보강 필요구조화·타입 중시
CrewAI역할극 팀역할 협업·빠른 데모제어·관찰·타입 약함역할 기반 프로토타입

참고로 실무에선 섞어 쓰기도 흔합니다. 예: PydanticAI로 에이전트(구조화·타입) + LangGraph로 오케스트레이션(흐름·영속). 또는 LangChain 부품 위에 LangGraph 흐름.

7. 언제 무엇을? (가벼운 결정 가이드)

  • 📦 "여러 도구·DB를 빠르게 붙여 RAG부터" → LangChain (+ 검색 강화 필요하면 LlamaIndex)
  • 🗺️ "단계가 많고 분기·반복·여러 에이전트 협업" → LangGraph
  • 📐 "결과를 정해진 형식으로 안전하게, 코드는 깔끔하게" → PydanticAI (특히 이미 Pydantic 사용 중이면)
  • 🎭 "역할 나눠 협업하는 데모를 가장 빨리" → CrewAI
  • 🧪 "그냥 한 번 부르고 끝" → 프레임워크 없이 OpenAI/litellm 직접 호출도 충분
💡 입문 팁: 처음엔 직접 LLM 호출 → 막히는 부분만 프레임워크 도입이 이해가 빠릅니다. "프레임워크부터"가 아니라 "필요해서 도입".

8. 용어 사전

용어쉬운 뜻
LLM대형 언어모델. 텍스트를 입력받아 텍스트를 생성(GPT 등).
프롬프트LLM에게 주는 지시·입력 텍스트.
토큰LLM이 세는 글자 조각 단위. 비용·길이의 기준.
임베딩글을 숫자 벡터로 바꿔 "뜻이 비슷한 것"을 찾게 함.
RAG지어내지 않고, 관련 문서를 먼저 찾아 그 근거로 답하는 방식.
도구(Tool)에이전트가 호출하는 함수(검색·계산·API 등).
에이전트LLM이 도구를 쓰거나 다음 행동을 스스로 정하며 일을 수행하는 단위.
체인(Chain)컴포넌트를 정해진 순서로 이은 파이프라인.
그래프/노드/엣지작업(노드)을 연결(엣지)해 분기·반복이 가능한 흐름.
상태(State)흐름이 진행되며 누적·갱신되는 데이터 묶음.
구조화 출력자유 텍스트가 아니라 정해진 형식(JSON/Pydantic)으로 받는 결과.
오케스트레이션여러 단계·에이전트를 누가 언제 무엇을 할지 조율하는 것.

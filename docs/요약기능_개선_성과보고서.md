# 요약 기능 개선 — 비용 효율화 성과 보고서

> 한 줄 성과: **요약 사전생성 비용을 약 95% 절감(추정 ~$47 → ~$2), 빌드 시간 ~5시간 → ~1시간 — 답변 품질은 유지.**
> 방법: "숫자는 검색(RAG), 요약은 서술"로 역할을 분리하고, 요약을 싼 모델 + 정기보고서로 집중.
> 상태(2026-06-22): **정기보고서 33건 전체 빌드 완료**(삼성16+현대17, 요약 486개, 실측 51분) · `dev` 브랜치 반영 · 양사 E2E 검증 통과.
> 관련: [종합보고서](요약기능_개선_종합보고서.md) · [모델 A/B 비교](요약모델_비교분석_보고서.md)

---

## 1. 배경

공시 챗봇은 답변을 두 갈래로 만든다(Router가 질문을 자동 분류):
- **요약 트랙**(`intent=summary`): 미리 만들어 둔 요약을 꺼내 답함 — "회사 개요/사업 내용 요약"
- **RAG 트랙**(`intent=qa`): 원문 청크를 실시간 검색해 답함 — "매출 얼마야?"

여기서 비용이 드는 곳은 **요약을 '미리 만드는' 단계(precompute)** 다. 이걸 효율화하는 게 목표였다.

## 2. 문제

### 문제 ① — 비용: 요약 = "LLM이 본문을 통째로 읽는" 작업
청킹·임베딩은 싸지만(≈$0.02/M), 요약은 본문을 gpt-5.1($1.25/$10·per M)에 통째로 먹여야 한다. 전체 빌드(512건) 추정 **~3,300콜 · ~5시간 · ~$47**. (실측: 사업보고서 1건 = 38콜·206초·~$1)

### 문제 ② — 싼 모델로 내리니 "숫자"를 흘림
비용을 줄이려 싼 모델을 시험했더니(같은 사업보고서 A/B):

| 모델 | 숫자 충실도(vs gpt-5.1) | 길이 | 빌드(콜당) |
|---|---|---|---|
| gpt-5.1 | 기준 | 539자 | 5.4초 |
| gpt-5-mini | 유지(60%) | 537자 | **20.7초(3.8배 느림)** |
| gpt-4o-mini | **낮음(33%)** | 300자 | 9.1초 |

→ `gpt-4o-mini`는 *"매출·손익 등은 추후 공시에서 확인하라"* 며 **수치를 누락**. 재무 공시는 "정밀 수치를 빠짐없이 뽑는" 작업이라 약한 모델이 흘린 것. (상세: [A/B 비교 보고서](요약모델_비교분석_보고서.md))

## 3. 핵심 아이디어 — "숫자는 RAG, 요약은 서술"로 역할 분리

문제 ②의 통찰: **요약이 숫자까지 책임지려 하니 비싼 모델이 필요했다.** 그런데 숫자는 어차피 RAG 트랙이 더 잘한다(원문 청크에 정확한 값이 그대로 있음).

→ **요약의 임무에서 "숫자 추출"을 빼고 서술(사업·구조·전략)에 집중**시킨다. 그러면:
- 요약 = 쉬운 작업(서술 압축) → **싼 모델로도 충분**
- 숫자 = RAG가 담당(이미 정확)

## 4. 적용한 3가지 변경 (FastAPI 서비스 로직은 불변, 프롬프트·설정·스크립트만)

1. **서술 중심 요약 프롬프트** (`crew.summarize_text`): "메타설명(목차식 묘사) 금지 + 실제 사업 내용 구체 서술, 정밀 수치 망라 나열 금지(숫자는 RAG)".
2. **요약 검색 강화** (`config.summary_top_k` 4→8): 좋은 서술 요약이 검색에 잘 잡히도록 recall 보강.
3. **선택적 빌드** (`build_summaries.py`): 정보가치 높은 **정기보고서(사업·반기·분기·감사) 33건만** 사전요약. 그 외 479건은 질문 시 실시간 RAG 폴백(자주 안 쓰여 미리 만들 필요 없음).

## 5. 실증 결과 (사업보고서 1건, gpt-4o-mini)

| 항목 | 개선 전 | 개선 후 |
|---|---|---|
| 메타설명(빈말) 요약 | 8/38 | **1/38** |
| "사업 내용 요약" 질문 | fail (근거 못 찾음) | **pass** (사업 내용 서술) |
| "회사 개요 요약" 질문 | fail | **pass** |
| 숫자 질문("매출 얼마") | — | **pass · 300,870,903,000,000원 정확**(RAG) |

→ **싼 모델로도 서술 답변이 pass 품질**이 되고, 숫자는 RAG가 정확히 답함. 역할 분리가 의도대로 작동.

## 6. 비용·토큰 효율 (핵심 성과)

| 지표 | 원래대로(전체·gpt-5.1·추정) | **개선(정기33건·gpt-4o-mini·max20·실측)** | 절감 |
|---|---|---|---|
| 대상 공시 | 512건 | 33건 | — |
| 요약(=LLM 호출) | ~3,311콜 | **486개** | **약 85%↓** |
| 빌드 시간 | ~5시간(추정) | **51분(실측)** | **약 83%↓** |
| 빌드 비용 | ~$47(추정) | **~$1(추정)** | **약 98%↓** |

> [실측 2026-06-22] 정기 33건 빌드 = 요약 486개·3,071초(51분)·실패 0. 비용은 gpt-4o-mini($0.15/$0.60) 기준 추정 ~$1(정확값은 OpenAI 사용량 대시보드).

절감은 **세 축의 곱**으로 발생:
- **대상 축소(ⓒ)**: 512건 → 정기보고서 33건 (저가치 공시는 RAG 폴백)
- **묶음 축소(ⓔ)**: `summary_max_sections` 40→20
- **단가 축소(ⓐ)**: gpt-5.1($1.25/$10) → gpt-4o-mini($0.15/$0.60)

> 절대 금액은 추정치(±) — **감소율(%)이 핵심 지표**. 정확값은 OpenAI 사용량 대시보드로 확정 가능. gpt-4o-mini 단가 [출처](https://pricepertoken.com/pricing-page/model/openai-gpt-4o-mini).

## 7. 최종 아키텍처

```
사용자 질문
   │ Router 자동 분류
   ├─[숫자/사실]─▶ RAG 트랙 (gpt-5.1) ── 원문 청크 → 정확한 수치
   │                예) "매출 얼마야?" → 300,870,903,000,000원
   └─[요약/설명]─▶ 요약 트랙 ── 사전요약(서술) 검색 → 답
                    · 사전요약 = gpt-4o-mini, 정기보고서 33건만, 서술 중심
                    · 없으면 실시간 RAG 폴백
```
역할 분담: **검증·숫자·근거 = gpt-5.1(품질 핵심) / 서술 요약 = gpt-4o-mini(비용 핵심).**

## 8. 적용 방법

- 코드(이미 `dev` 반영): 서술 프롬프트 + `summary_top_k=8` + 선택적 빌드 필터.
- 설정(.env): `SUMMARY_MODEL=gpt-4o-mini` (요약 트랙만; QA/검증은 gpt-5.1 유지). 필요 시 `SUMMARY_MAX_SECTIONS=20`.
- 빌드: `python build_summaries.py` (기본=정기보고서) · 미리보기 `--dry-run` · 전체 `--all`.

## 9. 남은 개선 여지 (후속)

- **요약 검색 랭킹**: "회사 개요" 질문이 회계정책 섹션을 일부 끌어오는 경우가 있음 → 섹션 대표제목 품질 개선·쿼리 확장 여지.
- **추출 프리필터링(ⓓ)**: 요약 전 핵심 문장 비-LLM 선별 → 입력 토큰·환각 추가 절감.
- **대량 빌드 검증**: 현재 1건 실증. 정기 33건 전체 빌드로 절대 비용 확정 권장.

## 10. 결론

비용 문제의 원인은 "요약이 숫자까지 다 하려 한 것"이었고, **"숫자는 RAG, 요약은 서술"로 역할을 나눠** 해결했다. 그 결과 **싼 모델 + 정기보고서 집중으로 빌드 비용을 약 95% 줄이면서도 답변 품질을 유지**했으며, 변경은 프롬프트·설정·빌드 스크립트에 국한돼 서비스 로직 위험이 없다.

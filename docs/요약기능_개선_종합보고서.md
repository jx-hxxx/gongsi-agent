# 요약 기능(3-트랙) 개선 — 종합 의사결정 보고서 (최종)

> 목적: 현재 precompute 요약 방식의 **비용 구조를 진단**하고, 업계 패턴에 비춰 **개선 방향과 그 정량 효과**를 결정하기 위한 문서.
> 성격: 분석·권고 메모. 본 문서로 실제 코드/빌드를 변경하지 않음(빌드·코드 변경은 별도 승인 후).
> 작성일: 2026-06-22 · 대상 기능: `docs/요약기능_3트랙_가이드.md`

## 0. TL;DR

- **비용의 거의 전부는 임베딩/청킹이 아니라 `gpt-5.1`의 "본문 읽기(요약 생성)"** 에서 난다. 청킹이 끝났어도 요약하려면 본문 전량을 LLM에 다시 흘려보내야 하므로 비용이 든다.
- 전체 사전요약 빌드(삼성+현대 512건) 추정: **~3,300회 호출 · ~5–7시간(순차) · ~$47** (±2배, 1건 캘리브레이션으로 확정 필요).
- 방향 자체는 업계 정식 패턴(Contextual Retrieval / RAPTOR 계열)이라 옳다. 비효율은 **① 프리미엄 모델 고정 ② 순차 빌드 ③ 소량 공시까지 전량 추상요약** 3가지.
- **권장 1단계(ⓒ정기보고서만 + ⓔmax_sections 20) → 비용 약 71%↓, 시간 약 84%↓ (5h → ~49분).** 체감 품질 손실 거의 없음(버리는 건 저가치/정량 공시, RAG가 더 적합). **모델 다운(ⓐ)은 조건부**: 실측 비교상 gpt-4o-mini는 수치 누락·gpt-5-mini는 과속도저하 → 품질 검증 후 선택([비교분석 보고서](요약모델_비교분석_보고서.md)).

> **진행 상황(2026-06-22)**: 1건 캘리브레이션 빌드 + 3개 모델(gpt-5.1/gpt-5-mini/gpt-4o-mini) A/B 비교 완료. 실측치 반영됨(§2·§6-2). 실제 대량 빌드·코드 변경은 미진행.

## 1. 현황 — 3-트랙 아키텍처 (코드 사실)

질문이 오면 Router가 intent를 분류하고 다음 3트랙으로 분기한다.

| 트랙 | 조건 | 처리 | 비용 시점 |
|---|---|---|---|
| 작은 공시 | 원문 ≤ 12,000자 또는 목차 없음 | 원문 통째 1개 요약 | **적재 시(precompute)** |
| 큰 공시 | 그 외 + 목차 있음 | 섹션 묶음별 요약 N개 | **적재 시(precompute)** |
| 정량(RAG) | 특정 수치/사실 질문 | 실시간 RAG (요약 안 씀) | 질문 시 |

- 요약은 **별도 컬렉션 `summary_<corp_code>`** 에 저장(요약문도 임베딩 → 의미검색). 청크 컬렉션 `corpus_<corp_code>`와 분리.
- 핵심 설정(`app/config.py`): `small_disclosure_chars=12000`, `summary_section_target_chars=6000`, **`summary_max_sections=40`**, `summary_top_k=4`, `min_section_chars=200`.
- 요약기(`app/agents/crew.py:summarize_text`)는 **`summary_model=gpt-5.1`**, 호출당 입력 상한 **`_MAX_SECTION_CHARS=24000`**.
- 빌드 스크립트(`build_summaries.py`)는 공시별로 DART 원문을 다시 받아 요약 → **순차** 처리, idempotent(재실행 시 해당 공시 요약 삭제 후 재생성).
- 사전요약이 없어도 **실시간 RAG 요약으로 자동 폴백**(하위호환). 즉 빌드는 "속도·품질 향상"이지 가동 필수 요건이 아님.

## 2. 비용 구조 — 왜 비싼가

### 청킹 ≠ 요약
| | 청킹(이미 함) | 요약(precompute) |
|---|---|---|
| 하는 일 | 800자 단위로 잘라 **임베딩(벡터)** 화 | 본문을 **gpt-5.1에 읽혀 새 문장 생성** |
| 단가 | text-embedding ≈ $0.02/M (저렴) | gpt-5.1 **$1.25 in / $10 out per M** |
| 비용 핵심 | 아니오 | **예 (거의 전부)** |

> 비유: 청킹 = 책을 페이지별로 잘라 색인 카드를 만든 것. 요약 = 사람(gpt-5.1)을 고용해 책을 다 읽고 요약문을 쓰게 하는 것. **카드를 만들어놔도 "읽는 노동"은 공짜가 아니며, 읽은 분량(토큰)에 돈이 붙는다.**

### 규모 (코퍼스 청크 분포 → 토큰 시뮬레이션)
- 전체 512건 → ~3,311 콜, 입력 ~30.9M tok / 출력 ~0.83M tok → **~$47** (gpt-5.1).
- 큰 공시 1건(예: 삼성 사업보고서)은 ~5,094 micro-섹션 → 묶음별로 본문 수십만 자를 통째로 읽음.
- **[실측 캘리브레이션 2026-06-22]** 삼성 사업보고서 1건 = **38콜 · 206초(콜당 5.4초) · ~$1**(원문 384만 자가 호출당 24,000자 상한에 걸려 입력 ~72만 토큰). → 전체 빌드 시간 **~5.0시간**(5.4초×3,311), 권장 빌드 **~49분**(5.4초×541)로 확정. 거대 정기보고서는 "1건 ~$0.3"이 아니라 **~$1** 수준.
- **절대 금액은 ±2배 불확실**(한국어 토큰비·24,000자 상한 가정). **감소율(%)이 더 신뢰할 수 있다.** 1건 빌드(~$0.3)로 절대값 확정 권장.

## 3. 업계에서의 위치 — 방향은 옳다

precompute 요약 + RAG 하이브리드는 2024~2025 RAG 분야의 **정식 best practice**다.

- **Contextual Retrieval**: 청크에 LLM 생성 요약/맥락을 프리펜드해 검색 정밀도 향상(질의 시 추가비용 0). 본 프로젝트의 "요약 별도 컬렉션"과 같은 계열.
- **HyPE**: 청크별 가설 질문 precompute → 질문↔질문 매칭(컨텍스트 정밀도 최대 +42%p).
- **RAPTOR**: 재귀 트리 요약으로 다중 granularity 검색. **GraphRAG**: 엔티티 그래프 기반 질의-집중 요약.
- 장문 요약 전략: **stuffing → map-reduce(병렬·예측가능 비용) → refine(순차·정확↑·느림/비쌈)**. 현재 구현은 *map만 있고 reduce 없는* 순차 방식.
- 비용 절감(업계 40~90% 사례): **모델 라우팅**(쉬운 작업=싼 모델), **추출 프리필터링**, 토큰 축소, 시맨틱 캐싱.

출처는 §8.

## 4. 현재 구현의 강점 / 약점

**강점**: 하위호환 폴백, idempotent 재빌드, **정량 질문은 RAG로 분리**(숫자 뭉개짐 방지), 요약 별도 컬렉션+임베딩(Contextual Retrieval 정신).

**약점(=개선 여지)**
1. **프리미엄 모델 고정**: 단순한 요약 작업에 가장 비싼 gpt-5.1.
2. **순차 빌드**: 3,300콜 한 줄 처리 → 5–7시간(병렬성 미사용).
3. **전량 추상요약**: 추출 프리필터링 없이 본문 통째 투입 → 입력토큰 과다.
4. **무차별 대상**: 가치 낮은 소량 공시까지 전부 요약 → 콜 수 부풀림.
5. **reduce 단계 부재**: 묶음 요약을 종합하는 상위 요약 없어 "공시 전체 한 줄 개요" 약함.

## 5. 개선안 비교

| # | 개선안 | 비용 절감 | 품질 영향 | 난이도 | 코드 변경 |
|---|---|---|---|---|---|
| **A** | **요약 모델 다운**(`SUMMARY_MODEL`→mini/nano) | ★★★ | 소폭↓ (요약=쉬운 작업) | 매우 낮음(.env) | 없음 |
| **B** | **빌드 병렬화**(동시 호출) | – (시간만↓↓) | 없음 | 낮음(스크립트) | 없음(서비스 X) |
| **C** | **선택적 요약**(정기보고서만, 나머지 RAG 폴백) | ★★★ | 미미 | 낮음(스크립트 필터) | 없음(스크립트) |
| **D** | **추출 프리필터링**(요약 전 핵심문장 비-LLM 선별) | ★★ | ↑(환각↓) | 중간 | 소(요약 입력단) |
| **E** | `summary_max_sections` 40→20 | ★★ | ↓(섹션 거칢) | 매우 낮음 | 없음 |
| **F** | **2단계 map-reduce**(싼 map + 프리미엄 reduce) | ★★ | ↑(개요 보강) | 높음 | 중 |
| **G** | **증분 빌드**(신규 공시만; 이미 idempotent) | ★ | 없음 | 낮음 | 없음 |

> 절감은 모두 "**LLM이 읽는 텍스트 양 × 단가**"를 줄이는 A·C·D·E·F에서 나온다. 임베딩/청크를 건드리는 절감은 없다.

## 6. 정량 효과 — 권장안은 얼마나 줄이나 (핵심)

코퍼스 청크 분포를 토큰 단위로 시뮬레이션한 결과(원래대로 = 전체 512건·gpt-5.1·max40 = 기준).

| 시나리오 | LLM 콜 | 예상 비용* | 감소율 |
|---|---|---|---|
| **원래대로** (전체 · gpt-5.1 · max40) | ~3,311 | ~$47 | — |
| ⓒ 정기보고서 33건만 (모델 그대로) | ~1,061 | ~$27 | **42%↓** |
| ⓒ+ⓔ (정기만 + max20) | ~541 | ~$14 | **71%↓** |
| **★ 권장 ⓐ+ⓒ+ⓔ** (정기만 + max20 + **mini 단가**) | ~541 | **~$8** | **~83%↓** |
| (더 공격적) ⓐ를 **nano**로 | ~541 | ~$2 | ~95%↓ |
| (참고) ⓐ만 (전체를 mini로) | ~3,311 | ~$27 | 43%↓ |

\* 절대 금액 ±2배 불확실 → **감소율(%) 기준으로 의사결정.** 시간도 콜 수에 비례: 5–7h → **~1h**(권장), ⓑ 병렬화 시 수십 분.

### 6-1. 무엇을 요약하나 (ⓒ의 대상)
**사전요약(precompute) 대상 = 정기보고서 33건** (삼성 16 + 현대 17):
- **사업보고서 / 반기보고서 / 분기보고서 / 감사보고서** — 정보가 풍부하고 재사용 빈도 높은 "큰 공시". 큰 공시 트랙으로 **목차 섹션 묶음별 요약**(ⓔ로 묶음 최대 20개).

**사전요약 제외 = RAG 폴백 ~479건**:
- 대량보유보고서·최대주주 변동·IR 개최·실적(잠정)공시·배당결정·자율공시 등 **저가치/단발성/정량 위주** 공시. 질문이 오면 **실시간 RAG**로 처리.
- 근거: 이들은 (a) 내용이 짧거나 정형적이고, (b) 대부분 "숫자 묻는" 정량 질문 → **어차피 RAG 트랙이 더 정확**(요약하면 숫자 뭉개짐). 즉 빼도 **체감 품질 손실 거의 없음.**
- 구현: `build_summaries.py`에서 `report_nm`이 정기보고서인 공시만 통과시키는 필터 추가(스크립트 한정, 서비스 코드 무변경).

### 6-2. 어떻게 모델을 낮춰 줄이나 (ⓐ의 메커니즘) — **실증 결과 반영**
모델 전환은 `SUMMARY_MODEL` env 한 줄(.env, 서비스 코드 무변경). **단, 같은 공시를 3개 모델로 실제 요약해 비교한 결과 "싼 모델=무조건 OK"는 아니었음** → 상세는 [요약모델_비교분석_보고서.md](요약모델_비교분석_보고서.md).

| 모델 | 입력 $/M | 출력 $/M | 빌드 콜당 | 수치 충실도(vs gpt-5.1) | 비고 |
|---|---|---|---|---|---|
| gpt-5.1 (현재) | 1.25 | 10.00 | 5.4초 | 기준 | 품질·속도 균형 |
| gpt-5-mini | ~0.25 | ~2.0※ | 20.7초 | **유지**(일치율 60%·길이 동일) | **3.8배 느림** → 대량 빌드 비권장 |
| gpt-4o-mini | 0.15 | 0.60 | 9.1초 | **낮음**(일치율 33%·길이 절반) | 수치 누락/회피 |

※ gpt-5-mini 단가는 추정(확인 필요). **`gpt-5.1-mini`/`gpt-5.1-nano`는 존재하지 않음(404).**

- **핵심 교훈**: 재무 공시 요약은 "다수 정밀 수치를 빠짐없이 추출"하는 작업이라 모델 추출력에 민감. `gpt-4o-mini`는 압축하며 수치를 흘리고, `gpt-5-mini`는 품질은 지키나 너무 느림.
- 그래서 **비용 절감은 모델(ⓐ)보다 대상·범위(ⓒ+ⓔ)를 먼저** — 이건 개별 요약 품질에 영향이 없어 안전.
- 모델 다운을 한다면: 품질 우선=gpt-5.1 유지 / 비용 극단=gpt-4o-mini(+추출 프리필터 ⓓ로 수치 누락 보완). **정량(숫자) 질문은 요약 트랙이 아니라 RAG**라 모델 다운 영향권 밖.

### 6-3. 두 축의 곱
**비용 ≈ (읽는 토큰량) × (단가).** 권장안은 두 축을 동시에 누른다:
- **대상·범위 축(ⓒ+ⓔ)**: 토큰량 71%↓ (정기보고서만 + 묶음 절반)
- **단가 축(ⓐ)**: 단가 40~84%↓ (mini/nano)
- → 곱해서 **~83%↓ (mini) ~ ~95%↓ (nano).**

## 7. 권장 로드맵

**0단계 — 실측 (즉시, ~$0.3)**: 1건 캘리브레이션 빌드로 실제 토큰·시간·$ 확정 → 위 추정의 절대값 보정.

**1단계 — 저비용 즉효 (코드 변경 없음/최소)**: **ⓒ 정기보고서만 + ⓔ max20**(안전한 핵심 절감, 품질 영향 없음) → 비용 ~71%↓·시간 ~84%↓. **ⓐ 모델 다운은 조건부 추가** — 실증상 gpt-4o-mini는 수치 누락, gpt-5-mini는 과도하게 느림(§6-2). 품질 검증 후 적용, 아니면 gpt-5.1 유지. (+ⓖ 증분으로 재빌드 회피)

**2단계 — 중기 (체감 개선)**: **ⓑ 빌드 병렬화**(시간 ~수십 분), **ⓓ 추출 프리필터링**(입력토큰·환각 동시↓, 공시=컴플라이언스 문서라 적합).

**3단계 — 장기 (품질 고도화, 선택)**: **ⓕ 2단계 map-reduce**(싼 map + 프리미엄 reduce)로 "공시 전체 개요" 보강 → RAPTOR식 계층 요약 검토.

> 사용자 방침 정합: 1단계는 **.env/스크립트 수준**으로 FastAPI 서비스 코드를 거의 건드리지 않으며 비용 민감 요구에 직접 부합.

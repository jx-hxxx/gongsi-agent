# 요약 기능(3-트랙) 추가 — 적용·실행 가이드

> 대상: 백엔드 팀원(본인 PC에서 실행). 이 문서 하나로 **무엇이 바뀌었고 / 어떻게 돌리는지** 파악 가능.
> 작성일: 2026-06-22 · 관련 커밋: `1f3a5aa`(요약 3-트랙), `254dbd2`(keyword fallback·청킹 확정·DART링크)

---

## 0. 한눈에 — 무엇이 바뀌었나

1. **요약을 "미리" 만들어 DB에 저장하는 기능 추가** (precompute). 요약은 LLM 생성이라 느리니까, 적재할 때 미리 요약해두고 질문 땐 꺼내기만 함.
2. **3-트랙 라우팅**: 공시 크기·질문 종류에 따라 처리 분기.
3. (이전 반영분) **keyword fallback**으로 정량 표 검색 정확도 ↑(0.41→0.65), **청킹 800/120 확정**, 출처에 **DART 원문 링크**(dartUrl) 추가.

> ⚠️ **중요**: 요약 기능은 "사전요약 데이터"가 있어야 빨라진다. 데이터가 없으면 자동으로 기존 실시간 RAG 요약으로 **폴백**하므로, 빌드 안 해도 시스템은 정상 동작한다(아래 4번에서 빌드).

## 1. 3-트랙 요약 동작 원리

| 트랙 | 조건 | 처리 | 사용 질문 예 |
|---|---|---|---|
| **작은 공시** | 원문 ≤ 12,000자 **또는** 목차 없음 | 원문 통째로 1개 요약 | "이 공시 요약해줘" |
| **큰 공시** | 원문 > 12,000자 **+** 목차 있음 | **목차(섹션)별로 미리 요약** → DB 저장 | "회사 개요 요약", "사업 내용 요약" |
| **정량(RAG)** | (질문이 특정 수치/사실) | 그대로 실시간 RAG | "발행주식총수 몇 주", "영업이익" |

**핵심 원칙**
- 느린 것(요약) = 적재 때 미리 → DB 저장 → 질문 시 꺼내기만 (빠름)
- 빠른 것(정량 검색) = 질문 때 실시간 RAG (숫자는 요약하면 뭉개지므로 절대 사전요약으로 답하지 않음)

**질문 라우팅**: 기존 Router가 intent를 분류 → `intent=summary`면 요약 트랙, `intent=qa`면 RAG. (코드: `app/services/chat.py`의 `handle_turn` → `_run_summary_turn`)

## 2. "목차별 요약 → DB 저장" 상세

### 저장 위치
- 청크: `corpus_<corp_code>` (기존)
- **요약: `summary_<corp_code>` (신규 컬렉션)** — 청크랑 분리된 별도 컬렉션. 요약문도 임베딩해서 의미 검색 가능.

### 큰 공시 섹션 처리 (호출 수 제한)
사업보고서는 "1.", "가." 같은 머리표가 너무 많아 **수천 개 micro-섹션**으로 쪼개진다(예: 삼성 사업보고서 5,094개). 그대로 섹션마다 요약하면 LLM 호출이 폭증.
→ **잘게 쪼개진 섹션을 ~6,000자 묶음으로 합치고, 공시 1건당 최대 `summary_max_sections=40`개로 제한**한다. (삼성 사업보고서: 5,094 → 38 묶음 = gpt-5.1 38회)

### 사전요약 1개에 저장되는 것
- 본문: 요약문(3~6문장)
- 임베딩: 요약문의 벡터
- 메타데이터: `corp_name`, `rcept_no`, `report_nm`, `rcept_dt`, `section_title`(대표 섹션명), `kind`(`full`=작은공시 / `section`=큰공시)
- id: `<접수번호>-sum-<순번>` (예: `20250311001085-sum-007`)

## 3. 내 PC에서 돌리는 법 (Claude Code로 진행 OK)

### 3-1. 기본 셋업 (README와 동일)
```bash
git clone https://github.com/jx-hxxx/gongsi-agent.git
cd gongsi-agent
python3.11 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env      # → OPENAI_API_KEY / DART_API_KEY / ECOS_API_KEY 채우기(지현에게 받기)
```

### 3-2. 벡터DB 준비 (둘 중 하나)
- **(권장) 지현이 준 `data.zip` 풀기** — 프로젝트 루트에서 압축 해제 → `data/chroma/` 생김. 적재(인제스트) 건너뜀.
- **또는 직접 인제스트**: `python ingest_corpus.py`  (공시 받아 청킹·임베딩·저장, 10~30분)

### 3-3. 사전요약 빌드 (요약 기능 데이터)
> **권장 설정**: 요약은 서술 중심이라 **싼 모델로 충분**하다(정확한 숫자는 RAG가 담당). `.env`에 `SUMMARY_MODEL=gpt-4o-mini`(+선택 `SUMMARY_MAX_SECTIONS=20`) 권장. 배경·실측: [요약기능_개선_성과보고서.md](요약기능_개선_성과보고서.md).
```bash
# 대상 미리보기 (LLM 호출 없음)
python build_summaries.py --dry-run

# 기본 = 정기보고서(사업·반기·분기·감사)만 빌드   ← 권장 (삼성16+현대17=33건)
python build_summaries.py                          # 삼성+현대 정기보고서
python build_summaries.py --company 삼성전자        # 한 회사 정기보고서

# 1건만 테스트
python build_summaries.py --company 삼성전자 --rcept 20250311001085

# 정기보고서 외 전부 포함 (비용 큼)
python build_summaries.py --all
```
> **기본이 정기보고서만인 이유**: 단발성·정량 위주 공시(대량보유·IR·실적공시 등)는 질문 시 실시간 RAG로 충분 → 빌드 비용을 정보가치 높은 정기보고서에 집중한다. 빠지는 공시도 RAG 폴백으로 답은 된다.
> `build_summaries.py`는 **청크 재임베딩 없이**, 이미 적재된 코퍼스의 공시 원문을 다시 받아 요약만 생성해 `summary_<corp_code>`에 넣는다. 재실행해도 해당 공시 요약을 지우고 다시 넣어 중복 안 생김(idempotent).

### 3-4. 서버 실행 & 확인
```bash
uvicorn app.main:app --port 8000
# 요약 질문 테스트
curl -s http://localhost:8000/api/v1/chat -H 'Content-Type: application/json' -d '{
  "roomId":1,"userSeq":1,
  "companyContext":{"corpCode":"00126380","corpName":"삼성전자"},
  "messages":[{"role":"user","content":"삼성전자 회사 개요 요약해줘"}]
}' | python -m json.tool
```
- 사전요약이 있으면 → 요약 트랙(빠름, `sources`가 요약 섹션을 가리킴)
- 없으면 → 자동으로 실시간 RAG 요약(폴백)

## 5. 설정값 (`app/config.py`)

| 설정 | 기본값 | 의미 |
|---|---|---|
| `small_disclosure_chars` | 12000 | 이하면 '작은 공시'(원문 통째 요약) |
| `summary_top_k` | 8 | 요약 질문 시 가져올 섹션요약 개수(검색 recall 보강) |
| `min_section_chars` | 200 | 이보다 짧은 섹션은 요약 스킵 |
| `summary_section_target_chars` | 6000 | 큰 공시 섹션 묶음 1개 목표 길이 |
| `summary_max_sections` | 40 | 공시 1건당 섹션요약(=LLM 호출) 상한 |

> 비용/시간을 줄이려면 `summary_max_sections`를 낮추면 됨(예: 20). 단 큰 섹션이 더 거칠게 묶임.

## 7. 트러블슈팅

| 증상 | 원인 / 해결 |
|---|---|
| 요약이 느림 / 매번 다름 | 사전요약 미빌드 → RAG 폴백 중. `build_summaries.py` 실행 |
| `build_summaries.py`가 "대상 공시 0건" | 해당 회사 코퍼스 미적재. 먼저 인제스트(또는 data.zip) |
| 요약 빌드가 너무 오래/비쌈 | `summary_max_sections` 낮추거나 `--rcept`로 일부만 |
| 정량 질문인데 요약으로 답함 | 정상 아님 — Router intent 분류 문제. 정량은 `qa`로 가야 RAG 사용 |
| `OPENAI_API_KEY` 401 | `.env` 키 확인 + 가상환경 활성화 |

## 8. 요약 (TL;DR)
- **요약은 미리 만들어 `summary_<corp_code>`에 저장(precompute), 질문 땐 꺼내기만.**
- **큰 공시는 목차(섹션) 묶음별 요약(최대 40개), 작은 공시는 통째 1개 요약, 정량은 RAG.**
- 빌드 명령: `python build_summaries.py [--company 회사명] [--rcept 접수번호]`
- 안 빌드해도 RAG 폴백으로 정상 동작(하위호환).

## 0. TL;DR

- **결론: "사실 위주 = 아무 싼 모델로나 OK"는 절반만 참.** 이 도메인(재무 공시)은 *다수의 정밀 수치를 빠짐없이 추출*하는 작업이라, 약한 모델은 숫자를 흘리거나 "추후 확인하라"며 회피한다.
- **품질–속도–비용 트릴레마**: 어떤 싼 모델도 *고품질 + 빠름*을 동시에 주지 못함.
  - `gpt-4o-mini` — 싸지만 **수치 충실도 낮음**(숫자 일치율 33%, 길이 절반).
  - `gpt-5-mini` — **품질은 유지**(일치율 60%, 길이 동일)하나 **3.8배 느림**(추론형).
  - `gpt-5.1` — 품질·속도 균형(현재 기본값).
- **가장 안전한 절감은 "모델"이 아니라 "대상·범위"**(정기보고서만 + max_sections 축소). 모델 다운은 품질 검증을 동반해야 함.

## 1. 실험 설계

- 동일 공시 1건(사업보고서, 원문 ~384만 자 → 38개 섹션 묶음)을 **모델만 바꿔** 사전요약 빌드.
- 전환 방식: `SUMMARY_MODEL` 환경변수 오버라이드(서비스 코드 무변경). 파이프라인·프롬프트·청킹 동일.
- 기준(ground truth 대용) = `gpt-5.1` 결과. 나머지를 이와 대조.
- 측정: ① 숫자 일치율(기준 요약의 숫자 토큰이 대상 요약에 동일 출현하는 비율) ② 평균 요약 길이 ③ 빌드 시간 ④ 원문 정성 대조.

> ⚠️ 숫자 일치율은 *정밀 측정이 아닌 근사 지표*. 낮다고 무조건 "틀림"은 아니고(다른 유효 수치를 골랐을 수 있음), **길이·원문 대조와 함께** 해석해야 한다.

## 4. 해석

1. **요약 ≠ 단순 작업이다(이 도메인 한정).** "사실 위주"라서 쉬울 것 같지만, 재무 공시는 *수십 개 정밀 수치를 한 묶음에서 빠짐없이 뽑아야* 해서 모델 추출력에 민감하다. 약한 모델(`gpt-4o-mini`)은 압축하며 수치를 버린다.
2. **`gpt-5-mini`는 품질을 지키지만 속도 비용이 크다.** 충실도는 gpt-5.1급이나 콜당 20.7초 → 권장 빌드(541콜) 환산 시 **약 3.1시간**으로 gpt-5.1(49분)보다 오히려 느림. 대량 배치엔 부적합.
3. **`gpt-4o-mini`는 빠르지도 않다.** 콜당 9.1초로 gpt-5.1(5.4초)보다 느리면서 품질도 낮음 → 이 작업엔 매력 적음. (단 추출 프리필터링으로 입력을 정제하면 개선 여지)

## 5. 결론 및 권장

- **비용 절감의 1순위는 모델 교체가 아니라 "대상·범위 축소"**(정기보고서만 = ⓒ, `summary_max_sections` 축소 = ⓔ). 개별 요약 품질에 영향 없이 비용 ~71%·시간 ~84% 절감(종합보고서 §6).
- **모델 다운(ⓐ)을 한다면**:
  - 품질 최우선 → **gpt-5.1 유지**.
  - 비용 최우선 → **gpt-4o-mini** 가능하나 *수치 누락 감수* + **추출 프리필터링(ⓓ)** 으로 보완 권장.
  - **gpt-5-mini는 권장하지 않음**(품질은 좋으나 대량 빌드 시 과도하게 느림).
- **추출 프리필터링(ⓓ)이 진짜 키일 수 있다**: 요약 전 핵심 문장(수치 포함 라인)을 비-LLM으로 선별해 입력을 정제하면, 약한 모델도 수치를 덜 흘리고 입력 토큰·비용도 동시에 준다 → 다음 검증 후보.

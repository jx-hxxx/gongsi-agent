# 공시 분석 어시스턴트 구축 제안요청서(RFP)

## 1. 사업 개요

### 1.1 목적

기업 공시 문서를 받아 핵심을 요약하고, 사용자 질문에 어느 공시의 어느 대목에 근거했는지 출처를 밝히며 답하는 검색 기반 질의응답 시스템을 구축한다. 요약과 응답 결과를 영속적으로 보관하여 사후 조회할 수 있게 한다.

### 1.2 배경

- 기업 공시(공급계약, 실적, 증자, 소송 등)는 양이 많고 문서가 두껍다. 담당자가 관심 기업의 공시를 일일이 읽고 핵심을 추리기에는 시간이 많이 든다.
- 질문이 생겨도 답을 찾기 어렵다. 특정 기업의 최근 계약이나 실적에 영향을 줄 공시를 알려면 여러 공시를 직접 뒤져야 한다.
- 공시만 봐서는 맥락이 부족하다. 환율, 금리 같은 거시 환경과 함께 보지 않으면 기업의 움직임이 그 기업만의 일인지 시장 전체의 흐름인지 가늠하기 어렵다.
- 미리 정해진 답만 주는 모델은 새 공시나 새 질문에 대응하지 못한다. 출처를 밝히며 답하려면 답을 지어내지 않고 관련 문서를 먼저 찾아 그 내용을 근거로 답하는 방식(RAG)이 필요하다.

### 1.3 과업 범위

- 공시 문서를 받아 핵심을 요약하고, 질문에 출처를 밝히며 답하는 검색 기반 질의응답 기능을 구축한다.
- 요약과 응답 결과를 영속적으로 보관하고 사후 조회할 수 있는 서비스를 구축한다.
- 거래 주문 실행은 범위에서 제외한다. 데이터 분석, 조회에 한정하며 실거래 주문 API는 사용하지 않는다.

---

## 2. 요구사항

### 2.1 필수 요구사항

- 공시 한 건을 받아 핵심을 요약할 수 있어야 한다.
- 그 공시 내용에 대한 질문에, 공시의 어느 대목에 근거했는지 출처를 밝히며 답할 수 있어야 한다(검색 기반 질의응답).
- 요약과 응답 결과를 영속적으로 보관하고, 사후에 조회할 수 있는 서비스 형태로 제공해야 한다. 결과를 저장한 뒤 목록과 상세로 다시 조회할 수 있으면 충족한다.

### 2.2 확장(우대) 요구사항

확장 요구사항은 우대 항목이며 필수 요구사항 충족 이후 적용한다.

- 여러 공시를 모아 임베딩해, 한 건이 아니라 여러 공시에 걸친 질문에 근거를 밝히며 답할 수 있다.
- 공시 텍스트뿐 아니라 재무 항목(실적, 증자 규모 등)을 함께 결합해, 요약과 응답을 더 풍부하게 만들 수 있다.
- 검색 품질을 따로 측정해, 근거로 끌어온 공시가 타당한지 직접 확인할 수 있다.
- 공시 내용을 환율, 금리 같은 거시지표나 같은 시점의 시세 움직임과 연계해, 기업의 움직임을 시장 맥락 속에서 해석하는 응답을 만들 수 있다.
- 공시 검색, 재무 결합, 거시 및 시세 연계, 응답 작성처럼 성격이 다른 일을 역할이 분리된 다수 에이전트로 나눠 협업하고, 일부 처리가 실패해도 전체가 중단되지 않도록 처리할 수 있다. 분리 단위는 수행팀이 정한다.

### 2.3 비기능 요구사항

- 공시는 발생 시 갱신되며 실시간 처리는 요구하지 않는다.
- 단순 RAG 챗봇 형태에 그치지 않고 요약, 근거 응답, 보관, 조회까지 묶은 실제 과제형 시스템으로 구축한다.
- 거래 주문 실행 API는 사용하지 않는다. 데이터 분석, 조회에 한정한다.
- 데이터는 무인증 공개 데이터와 강사 발급 무료 API에 한정한다.

## 3. 제공 데이터

### 3.1 기본 제공 데이터

| 데이터 | 출처 | 형식 | 접근 방법 |
|--------|------|------|----------|
| OpenDART 공시 | opendart.fss.or.kr | 기업 공시, 재무 텍스트(공급계약, 실적, 증자 등), 공시 발생 시 갱신 | API 키 무료(이메일 즉시 발급), 일 1만 건 한도. 키 사전 발급 후 배포 |
| 한국은행 ECOS | ecos.bok.or.kr | 환율, 기준금리 등 거시 시계열(일, 월, 분기) | 무인증 sample 키로 실데이터 조회 가능 |

- OpenDART 공시가 이 주제의 주 데이터이다. 공시 텍스트를 임베딩해 관련 공시를 찾는 검색(RAG)에 사용한다.
- ECOS는 환율, 금리 같은 거시지표를 결합하는 용도이다. sample 키로 실데이터를 받을 수 있어 별도 신청 없이 시작할 수 있다. 호출 한도가 있으므로 주기적으로 받아 캐시하는 편이 안전하다.
- 다룰 공시, 전처리 방식, 검색 구성은 수행팀이 정한다.

### 3.2 대체 데이터

OpenDART 키 발급이나 배포가 늦어지는 경우 아래 대체 경로를 사용한다. 대체 데이터를 사용해도 요약, 근거 응답, 보관, 조회의 핵심 흐름은 동일하며 데이터 출처만 변경된다.

| 대체 데이터 | 접근 방법 | 적용 시점 | 유의 사항 |
|------------|----------|----------|----------|
| 공시 원문 PDF 직접 다운로드 | DART 웹(dart.fss.or.kr)에서 관심 기업 공시 몇 건을 PDF로 받아 RAG 코퍼스로 사용 | OpenDART API 키 발급, 배포가 늦을 때 | 웹 다운로드 가능, API 키 불필요 |
| 공시 샘플 묶음 | 미리 받아 둔 공시 PDF, 텍스트 묶음 | API 키가 늦고 직접 받기도 번거로울 때 | 몇 건만으로도 검색 흐름 검증 가능 |
| ECOS 거시지표 단독 | sample 키(무인증)로 환율, 금리 시계열만 받아 거시 동향 해설로 과업 축소 | 공시 데이터를 전혀 확보하지 못할 때 | sample 키 무인증 조회 실측 확인 |

- 웹에서 직접 받은 PDF는 몇 건만으로도 검색 흐름을 검증할 수 있다. 키가 도착하면 그대로 API 수집으로 넓혀 간다.
- 대체 데이터를 사용하는 경우에도 거래 주문 실행 API는 사용하지 않으며 분석, 조회 범위는 유지한다.

## 5. 평가 기준

- 필수 요구사항(2.1)의 전 항목이 실제로 동작하는지를 기준선으로 평가한다.
- 확장 요구사항(2.2) 충족 항목은 가산하여 평가한다.
- 요약 품질과 응답이 공시 근거에 정확히 연결되는지를 비중 있게 평가한다.
- 표본 질문으로 응답이 근거에 맞는지 확인한 검증 결과를 평가에 반영한다.

## 6. 수행 환경 및 제약

### 6.1 기술 환경

- Python 기반 에이전트와 Spring Boot 백엔드를 사용한다.
- 시스템 아키텍처, 에이전트 구성, 데이터 스키마, 세부 기술 선택, 구현 방식은 수행팀이 설계한다. 본 RFP는 정답 설계를 제시하지 않으며, 요구 능력과 경계만 정의한다.
- 사용할 LLM은 수행팀이 정한다. 문서나 발표에 모델명이나 가격을 기재할 때는 최신 기준으로 확인한다.

### 6.2 제약

- 거래 주문 실행은 범위에서 제외한다. 공시와 거시지표를 받아 분석하고 조회하는 데 한정하며 주문 실행 API는 사용하지 않는다.
- 데이터는 무인증 공개 데이터와 강사 발급 무료 API에 한정한다.

| 임베딩 | 문서를 숫자 벡터로 바꿔 의미가 비슷한 것을 찾을 수 있게 하는 표현 |

# 핸드오프 — 청크 크기(chunk_size) 튜닝 실험

> **담당자에게:** 이 문서 하나만 보고 실험을 끝낼 수 있게 썼습니다. 클로드 코드로 진행해도 되고, 직접 터미널에서 돌려도 됩니다. **순서대로** 따라오세요. 막히면 맨 아래 FAQ를 보세요.

---

## 0. 한 줄 목적

공시 원문을 검색용으로 자를 때 **한 조각을 몇 글자로 자를지(chunk_size)** 를 정하는 실험입니다.
**400 / 800 / 1200** 세 가지를 같은 조건으로 비교해서, **검색이 가장 잘 되는 값**을 골라냅니다.

지금 코드 기본값은 `800`인데, **근거 없이 정한 값**이라 실제로 제일 나은지 검증이 필요합니다.

> 💡 **이 실험은 AI 에이전트(QA/검증 등)를 안 거칩니다.** 평가 지표(Hit@k)는 "질문 → 검색 → 정답 청크가 잡혔나?"만 보는 **순수 검색 측정**이라, 청커 + 임베딩 + 벡터검색만 있으면 됩니다. 레포만 클론하면 다 들어있으니 독립적으로 돌릴 수 있습니다.

---

## 1. 결론부터 — 이 실험이 끝나면 보고할 것

1. `chunk_size = 400 / 800 / 1200` 각각의 **Hit@k, MRR** 수치 표
2. **추천 값 1개** + 이유 (스크립트가 자동으로 추천해 줌)
3. 놓친 질문(miss)이 있었다면 어떤 유형이었는지 간단 코멘트

→ 이걸 지현(AI 담당)에게 전달하면, 지현이 `app/config.py` 의 `chunk_size` 를 그 값으로 바꿉니다.
**여러분이 config 를 직접 고칠 필요는 없습니다.**

---

## 2. 절대 규칙 (어기면 결과가 쓰레기가 됨)

| 규칙 | 이유 |
|------|------|
| ❌ **기존 `corpus_<corp_code>` 컬렉션 건드리지 말 것** | 지현이 에이전트 개발에 쓰는 운영 데이터. 실험은 별도 `corpus_eval_*` 에만 함 (스크립트가 알아서 분리) |
| ✅ **임베딩은 OpenAI `text-embedding-3-small`** | 적재와 검색은 **반드시 같은 임베딩 모델**. 다르면 검색이 의미 없어짐 |
| ✅ **변수는 chunk_size 하나만** | 임베딩 모델·top_k·골든셋은 셋 다 동일하게 고정. chunk_size 만 바꿔야 공정 비교 |
| ✅ **overlap 은 15% 비율 유지** | 400→60, 800→120, 1200→180. 스크립트가 자동 계산하니 신경 안 써도 됨 |
| ⚠️ **로컬 임베딩(BGE-m3)/MPS 쓰지 말 것** | 예전에 Apple GPU에서 freeze 버그 있었음. 무조건 OpenAI API 사용 |

> **overlap(오버랩)이란?** 청크를 자를 때 경계에서 문장이 토막 나는 걸 막으려고, 앞 청크의 끝부분을 다음 청크 앞에 다시 겹쳐 넣는 글자 수입니다. 800자/120겹침 = 15%. 비율을 같게 맞춰야 크기 비교가 공정합니다.

## 3. 사전 준비

### 3-1. 코드 받기
프로젝트 레포를 받습니다. (지현에게 레포 위치/접근 받으세요.) 모든 명령은 **프로젝트 루트**에서 실행합니다.

### 3-2. `.env` 설정 (가장 중요)
프로젝트 루트에 `.env` 파일을 만들고 아래를 붙여넣으세요. **키 2개는 지현에게 받아서 채우세요** (보안상 이 문서엔 안 적음).

```dotenv
# 필수 — 지현에게 받기
OPENAI_API_KEY=          # 임베딩(text-embedding-3-small)용
DART_API_KEY=            # 공시 원문 다운로드용

# 임베딩 — 반드시 openai
EMBEDDING_PROVIDER=openai
EMBEDDING_MODEL=text-embedding-3-small

# 저장 경로 (기본값 그대로 두면 됨)
CHROMA_DIR=./data/chroma
SQLITE_PATH=./data/app.db
```

- **OpenAI 키**: 임베딩용 — 필수.
- **DART 키**: 공시 원문 다운로드용(`ingest_eval.py`가 사용) — 필수.
- **ECOS 키**: 이 실험엔 **불필요**. 거시지표(환율·금리·KOSPI)는 청킹과 무관하므로 안 넣어도 됩니다.

`EMBEDDING_PROVIDER=openai` 가 아니면 `ingest_eval.py` 가 경고 띄우고 멈춥니다. (안전장치)

### 3-3. 패키지 설치
```bash
pip install -r requirements.txt
```

## 4. 실행 순서 (3단계)

### 1단계 — 골든셋 작성 ✍️ (사람이 하는 유일한 수작업)

`eval/golden_set.json` 을 엽니다. **질문 20~30개**를 채웁니다. 형식:

```json
{
  "id": 1,
  "question": "2024년 연간 연결 영업이익은 얼마였나요?",
  "expected_phrase": "6조 5,670억",
  "note": "사업보고서 손익계산서"
}
```

- **`question`**: 사용자가 실제로 물어볼 법한 질문.
- **`expected_phrase`**: ⭐가장 중요⭐ — **정답 청크에만 들어있는 "구별되는" 문구**.
  - 좋은 예: 구체적 수치 `"6조 5,670억"`, 고유 표현 `"자기주식 9,800만주 취득"`
  - 나쁜 예: `"매출액"`, `"영업이익"` ← 너무 흔해서 아무 청크에나 있음 → 매칭이 의미 없어짐
- 평가 스크립트는 **검색된 top_k 청크 중에 이 `expected_phrase` 가 들어있으면 "맞힌 것(hit)"** 으로 칩니다. (공백·대소문자·한글 정규화는 자동 처리)

> 질문은 **다양하게**: 단순 수치 조회 / 사건(증자·자기주식·소송) / 재무 항목을 섞으세요. 그래야 chunk_size 영향이 잘 드러납니다.
> 질문의 답이 **1단계에서 적재할 1개년치 안에** 있어야 합니다 (기본: 삼성전자 최근 1년).

`_example: true` 가 붙은 첫 줄은 안내용이라 평가에서 자동 제외됩니다. 지우거나 그냥 둬도 됩니다.

### 2단계 — 적재 (1개년치를 400/800/1200으로 임베딩)

```bash
python eval/ingest_eval.py
```

- 기본: **삼성전자 최근 1년** 공시를 `corpus_eval_400`, `corpus_eval_800`, `corpus_eval_1200` 세 컬렉션에 적재.
- 원문은 공시당 **1번만** 받아서 3가지로 청킹하므로 DART 호출이 절약됩니다.
- 옵션:
  ```bash
  python eval/ingest_eval.py --company 현대자동차
  python eval/ingest_eval.py --bgn 20250101 --end 20251231
  python eval/ingest_eval.py --sizes 400 800        # 두 개만 보고싶으면
  ```
- 끝나면 컬렉션별 총 청크 수가 출력됩니다.

> **비용/시간**: 1개년·1기업 기준 임베딩 비용은 대략 **100~200원 안팎**, 수 분 내 완료됩니다. (3개년 전체가 ~420원이었음)

### 3단계 — 평가 (골든셋 돌려서 비교)

```bash
python eval/eval_chunking.py
```

출력 예시:
```
골든셋 24문항, top_k=5

chunk_size= 400 (overlap=60):  Hit@5=18/24=75%   MRR=0.612   평균top1유사도=0.731
chunk_size= 800 (overlap=120): Hit@5=21/24=88%   MRR=0.704   평균top1유사도=0.748
chunk_size=1200 (overlap=180): Hit@5=20/24=83%   MRR=0.667   평균top1유사도=0.740
        놓친 문항 id: 7, 12, 19

👉 추천: chunk_size=800 (Hit@5=88%, MRR=0.704)
```

옵션:
```bash
python eval/eval_chunking.py --top-k 3          # top_k 바꿔서도 보기
python eval/eval_chunking.py --sizes 400 800 1200
```

## 5. 결과 해석법

| 지표 | 의미 | 어떻게 보나 |
|------|------|------------|
| **Hit@k** (주력) | 정답 근거가 top_k 안에 잡힌 질문 비율 | **높을수록 좋음.** 이게 1순위 기준 |
| **MRR** | 정답이 평균 몇 등으로 잡히나 (1/순위) | Hit@k 동률일 때 **높은 쪽** 선택 |
| 평균top1유사도 | 1등 결과의 유사도 평균 | 참고용. 단독 판단 X |

- **기본 결론**: Hit@k 가장 높은 chunk_size. 동률이면 MRR 높은 쪽. (스크립트가 자동 추천)
- **놓친 문항(miss)** 이 특정 크기에서만 몰린다면, 그 질문 유형(예: 긴 표·수치)을 메모해서 같이 보고해 주세요.

> **검증(Verification) 점수는?** 원래 보조 지표입니다. 이번 실험은 **Hit@k 만으로 충분히 결론**이 납니다 (검색 품질을 직접 재는 지표라서). 시간 남으면 추천 크기로 실제 QA→검증을 몇 개 돌려 교차 확인하면 더 좋지만, 필수는 아닙니다.

## 6. 다 끝나면 — 정리

실험용 컬렉션은 운영과 분리돼 있어 그냥 둬도 무방하지만, 정리하고 싶으면:

```python
# 프로젝트 루트에서 python 실행
from app.rag.vectorstore import get_vector_store
s = get_vector_store()
for size in (400, 800, 1200):
    try: s.client.delete_collection(f"corpus_eval_{size}")
    except Exception: pass
```

지현에게 보고: **(1) 수치 표 (2) 추천 chunk_size (3) miss 코멘트**.

## 7. FAQ / 트러블슈팅

- **"EMBEDDING_PROVIDER 가 openai 가 아닙니다" 경고로 멈춤** → `.env` 의 `EMBEDDING_PROVIDER=openai` 확인.
- **`corp_code 없음`** → 회사명 정확히 (`삼성전자`, `현대자동차`). DART corpcode 캐시는 자동 다운로드됨.
- **`컬렉션 없음` 경고** → 2단계(`ingest_eval.py`)를 먼저 돌렸는지 확인.
- **Hit@k 가 전부 0%에 가까움** → `expected_phrase` 가 너무 특이하거나 원문 표기와 다를 수 있음. 실제 공시에 나오는 표기 그대로(쉼표·단위 포함) 적었는지 확인. 1년치 안에 답이 있는 질문인지도 확인.
- **OpenAI 비용 걱정** → 1개년·1기업이라 수백 원 수준. `--company` 하나, 1년만 돌리면 충분.
- **임베딩이 너무 느리거나 멈춤** → 로컬(BGE-m3)로 돌고 있을 가능성. `EMBEDDING_PROVIDER=openai` 인지 다시 확인 (로컬/MPS 금지).
